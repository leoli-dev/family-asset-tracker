# Family Asset Tracker — Architecture & Deployment Guide

## 1. Overview

This document describes the migration from a pure frontend PWA (localStorage) to a self-hosted FE-BE-DB architecture deployable on a homelab Linux server.

**Before:** React SPA → localStorage  
**After:** React SPA → FastAPI (Python) → SQLite

---

## 2. Architecture

```
┌─────────────────────────────────────────┐
│  Browser                                │
│  React PWA                              │
│  - token stored in localStorage         │
│  - all state fetched from API on load   │
│  - every mutation is an API call        │
└──────────────┬──────────────────────────┘
               │ HTTP (Bearer token)
               ▼
┌─────────────────────────────────────────┐
│  FastAPI (Python, uvicorn)              │
│  - validates Bearer token vs .env       │
│  - serves /api/* routes                 │
│  - serves dist/ static files (SPA)      │
│  - single process, single port          │
└──────────────┬──────────────────────────┘
               │ sqlite3 (stdlib)
               ▼
┌─────────────────────────────────────────┐
│  SQLite                                 │
│  /opt/asset-tracker/data/db.sqlite      │
│  single file, backup = cp               │
└─────────────────────────────────────────┘
```

### Key decisions

- **No CORS:** BE serves the frontend static files directly. Same origin, no CORS headers needed.
- **No Docker:** Target is a minimal Debian VM. Python is pre-installed; only two pip packages needed.
- **No session/cookie:** Token lives in `localStorage` (consistent with existing PWA pattern). Sent as `Authorization: Bearer <token>` on every request.
- **Same token for UI and API:** The token in `.env` is the single secret. The login page just validates it client-side by making a test API call; there is no separate session issuance.
- **Settings in DB:** `defaultCurrency`, `language`, `logoUrl` move from localStorage to a `settings` key-value table so they persist server-side.

## 4. SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS settings (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- keys: default_currency, language, logo_url

CREATE TABLE IF NOT EXISTS owners (
    id   TEXT PRIMARY KEY,
    name TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS categories (
    id    TEXT PRIMARY KEY,
    name  TEXT NOT NULL,
    type  TEXT NOT NULL CHECK(type IN ('ASSET', 'LIABILITY')),
    color TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS accounts (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    currency    TEXT NOT NULL,
    category_id TEXT NOT NULL REFERENCES categories(id),
    owner_id    TEXT NOT NULL REFERENCES owners(id)
);

CREATE TABLE IF NOT EXISTS records (
    id         TEXT PRIMARY KEY,
    date       TEXT NOT NULL,     -- YYYY-MM-DD
    account_id TEXT NOT NULL REFERENCES accounts(id),
    amount     REAL NOT NULL,
    note       TEXT,
    timestamp  INTEGER NOT NULL
);
```

IDs remain UUID strings (generated by the frontend, consistent with existing data).

## 5. API Specification

All endpoints are prefixed `/api`. All require `Authorization: Bearer <token>` header.  
Returns `401` if token is missing or wrong. Returns `404` if resource not found.

### Auth

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/auth/verify` | Returns `200 OK` if token is valid. Used by login page. |

### Settings

| Method | Path | Body | Description |
|--------|------|------|-------------|
| GET | `/api/settings` | — | Returns `{defaultCurrency, language, logoUrl}` |
| PUT | `/api/settings` | `{defaultCurrency?, language?, logoUrl?}` | Partial update, returns updated settings |

### Owners

| Method | Path | Body | Description |
|--------|------|------|-------------|
| GET | `/api/owners` | — | List all owners |
| POST | `/api/owners` | `{id, name}` | Create owner |
| PUT | `/api/owners/{id}` | `{name}` | Update owner |
| DELETE | `/api/owners/{id}` | — | Delete owner (fails if referenced by accounts) |

### Categories

| Method | Path | Body | Description |
|--------|------|------|-------------|
| GET | `/api/categories` | — | List all categories |
| POST | `/api/categories` | `{id, name, type, color}` | Create category |
| PUT | `/api/categories/{id}` | `{name?, type?, color?}` | Update category |
| DELETE | `/api/categories/{id}` | — | Delete category (fails if referenced by accounts) |

### Accounts

| Method | Path | Body | Description |
|--------|------|------|-------------|
| GET | `/api/accounts` | — | List all accounts |
| POST | `/api/accounts` | `{id, name, currency, categoryId, ownerId}` | Create account |
| PUT | `/api/accounts/{id}` | `{name?, currency?, categoryId?, ownerId?}` | Update account |
| DELETE | `/api/accounts/{id}` | — | Delete account (fails if has records) |

### Records

| Method | Path | Body / Query | Description |
|--------|------|------|-------------|
| GET | `/api/records` | `?accountId=&from=&to=` | List records, all filters optional |
| POST | `/api/records` | `{id, date, accountId, amount, note?, timestamp}` | Create record |
| PUT | `/api/records/{id}` | `{date?, accountId?, amount?, note?, timestamp?}` | Update record |
| DELETE | `/api/records/{id}` | — | Delete record |

### Data Management (backup / restore / export)

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/backup` | Returns full `FullBackup` JSON (same schema as current export) |
| POST | `/api/restore` | Accepts `FullBackup` JSON, **replaces all data** (owners, categories, accounts, records) |
| GET | `/api/export/csv` | Returns CSV file download (same format as current CSV export) |

The `/api/backup` and `/api/restore` endpoints preserve full compatibility with the existing `FullBackup` JSON format, so current backup files can be used for migration.

## 6. Authentication Flow

```
1. User opens the app → React checks localStorage for 'fat_token'
2. If no token → redirect to /login page
3. User enters token → FE calls GET /api/auth/verify with Bearer header
4. 200 → store token in localStorage, redirect to dashboard
5. 401 → show error, clear token
6. On every subsequent API call → include Authorization: Bearer <token>
7. If any API call returns 401 → clear token, redirect to /login
```

The token in `.env` is a static string. No expiry, no rotation — appropriate for homelab internal use.

## 7. Frontend Changes

Changes to the existing React codebase:

1. **Add `LoginPage` component** — simple single-field form that calls `/api/auth/verify`
2. **Add `api.ts` service** — thin wrapper around `fetch` that injects the Bearer token and handles 401 globally
3. **Replace all localStorage reads/writes** with API calls in `App.tsx`:
   - `useEffect` on mount → `GET /api/settings`, `GET /api/owners`, etc.
   - All `handleSave*` / `handleDelete*` handlers → corresponding `POST`/`PUT`/`DELETE` calls
4. **Settings page** — export/import buttons call `/api/export/csv` and `/api/restore` instead of client-side logic
5. **Remove demo mode** — demo mode was a localStorage-era concept; with a real backend it is not needed
6. `logoUrl` from settings: store as base64 string in the settings table (same as current localStorage approach)

What does **not** change:
- All component UI code
- All translation logic
- CSV export format
- JSON backup/restore format

## 8. Data Migration (localStorage → SQLite)

Migration is one-time and uses the existing export/import flow:

**Step 1 — Export from current browser:**  
Settings → Backup JSON → saves `fat_full_backup_YYYY-MM-DD.json`

**Step 2 — Deploy the new server** (see Section 9)

**Step 3 — Import into new server:**  
Login → Settings → Restore → select the backup JSON file  
This calls `POST /api/restore` which loads all data into SQLite.

**Step 4 — Verify, then clear browser localStorage:**  
Once data is confirmed on the server, the old `fat_*` localStorage keys are no longer used.

## 9. Deployment — Blank Debian 13 VM

### 9.1 Server prerequisites

Debian 13 ships with Python 3.12. Verify:

```bash
python3 --version   # should be 3.12.x
```

Install pip if missing:

```bash
sudo apt update && sudo apt install -y python3-pip
```

### 9.2 Create app directory

```bash
sudo mkdir -p /opt/asset-tracker/data
sudo chown -R $USER:$USER /opt/asset-tracker
```

### 9.3 Copy files to server

Run from your dev machine:

```bash
# Build frontend first
cd family-asset-tracker/frontend
npm run build               # outputs to dist/

# Copy to server (replace SERVER_IP)
rsync -av dist/             user@SERVER_IP:/opt/asset-tracker/dist/
rsync -av backend/          user@SERVER_IP:/opt/asset-tracker/backend/
```

### 9.4 Install Python dependencies

On the server:

```bash
cd /opt/asset-tracker/backend
pip3 install -r requirements.txt
```

`requirements.txt` contains only:
```
fastapi==0.115.0
uvicorn==0.30.0
python-multipart==0.0.9
```

### 9.5 Configure environment

```bash
cat > /opt/asset-tracker/backend/.env << 'EOF'
ACCESS_TOKEN=your-secret-token-here
DB_PATH=/opt/asset-tracker/data/db.sqlite
STATIC_DIR=/opt/asset-tracker/dist
PORT=8000
EOF

chmod 600 /opt/asset-tracker/backend/.env
```

Generate a strong token:
```bash
python3 -c "import secrets; print(secrets.token_urlsafe(32))"
```

### 9.6 Create systemd service

```bash
sudo tee /etc/systemd/system/asset-tracker.service << 'EOF'
[Unit]
Description=Family Asset Tracker
After=network.target

[Service]
Type=simple
User=YOUR_USERNAME
WorkingDirectory=/opt/asset-tracker/backend
ExecStart=/usr/bin/python3 -m uvicorn main:app --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5
EnvironmentFile=/opt/asset-tracker/backend/.env

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable asset-tracker
sudo systemctl start asset-tracker
```

Check status:
```bash
sudo systemctl status asset-tracker
journalctl -u asset-tracker -f   # live logs
```

### 9.7 Access

Navigate to `http://SERVER_IP:8000` from any browser on the LAN.

If you want port 80 instead of 8000, either:
- Change `PORT=80` in `.env` and run as root (not recommended), or
- Use `authbind` to allow port 80 for non-root, or
- Add a simple nginx reverse proxy (one extra apt package)

## 10. Backup

SQLite backup is a single file copy:

```bash
cp /opt/asset-tracker/data/db.sqlite ~/backups/asset-tracker-$(date +%Y%m%d).sqlite
```

Set up a daily cron:

```bash
crontab -e
# Add:
0 2 * * * cp /opt/asset-tracker/data/db.sqlite ~/backups/asset-tracker-$(date +\%Y\%m\%d).sqlite
```

The Settings → Backup JSON in the UI also works as a human-readable backup and is compatible with the legacy browser-based app.

## 11. Updating the App

```bash
# On dev machine: rebuild frontend
cd frontend && npm run build

# Sync to server
rsync -av dist/    user@SERVER_IP:/opt/asset-tracker/dist/
rsync -av backend/ user@SERVER_IP:/opt/asset-tracker/backend/

# On server: restart
sudo systemctl restart asset-tracker
```

The SQLite database is untouched by updates.
